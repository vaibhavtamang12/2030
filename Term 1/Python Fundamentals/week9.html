1. The Performance Landscape

Performance engineering is the discipline of making software run faster or use fewer resources. For a quant engineer, this is not a luxury—it is a competitive advantage. A strategy backtest that takes 3 hours to run can only be iterated 3 times per day. An optimized backtest that takes 3 minutes can be iterated 180 times per day. That is the difference between finding a profitable strategy and missing it entirely.

1.1 Latency vs. Throughput

Performance has two faces. Imagine a coffee shop:

  • Latency is how long one customer waits from ordering to receiving their coffee. In trading, this is tick-to-trade latency—the time between seeing a market signal and executing an order. High-Frequency Trading (HFT) firms obsess over microseconds.
  • Throughput is how many customers the shop serves per hour. In quant research, this is how many backtests, parameter sweeps, or simulations you can run per hour. A data scientist cares more about throughput than latency.

Optimizing for one can hurt the other. A single espresso machine with an expert barista has low latency (each coffee is fast) but limited throughput. Ten automated vending machines have high throughput but slightly higher latency per cup. In code, batching work increases throughput but may add latency for individual items.

1.2 The Hardware Stack

Your code runs on physical hardware. Understanding the hierarchy of speed helps you make smarter decisions.

Figure 1.1: The Memory Hierarchy. Each level is ~10-100x slower than the one above it. Performance engineering is often about keeping data in the faster levels.

Notice the massive gap between RAM and disk. Reading from SSD is 100,000x slower than reading from CPU cache. When we optimize, we often try to:

  1. Keep data in cache (small, fast loops over contiguous memory)
  2. Avoid reading from disk (load data once, process in memory)
  3. Minimize RAM allocations (fewer objects = less garbage collection)

1.3 The Optimization Hierarchy

Not all optimizations are equal. Follow this priority order:

PriorityOptimizationTypical SpeedupDifficulty
1Algorithm & Big OInfinite (can make impossible possible)Medium
2Data Structures10x-100xLow
3Vectorization (NumPy/Pandas)10x-100xLow-Medium

Premature Optimization: Donald Knuth famously said, "Premature optimization is the root of all evil." This means: write clear, correct code first. Then measure. Then optimize the hot spots.

1.4 Colab Experiment: Feeling the Hardware

Let's write a small script to feel the difference between CPU-bound and memory-bound work.

Colab Experiment 1.1: Cache vs Memory Bound
import numpy as np
import time

# Small array fits in CPU cache
small = np.random.rand(1000)
# Large array exceeds cache, must live in RAM
large = np.random.rand(10_000_000)

def sum_array(arr):
    return np.sum(arr)

# Warm-up
sum_array(small)
sum_array(large)

# Time small (cache-friendly)
start = time.perf_counter()
for _ in range(10000):
    sum_array(small)
t_small = time.perf_counter() - start

# Time large (memory-bound)
start = time.perf_counter()
for _ in range(10):
    sum_array(large)
t_large = time.perf_counter() - start

print(f"Small (cache): {t_small*1000:.2f} ms total")
print(f"Large (RAM):   {t_large*1000:.2f} ms total")
print(f"Ratio per element: {(t_large/10)/(t_small/10000):.2f}x")

Expected Observation: The large array is not proportionally slower. It is much slower per element because the CPU constantly waits for RAM. This is the "memory wall."

2. Measuring Before Optimizing

The first rule of performance engineering is: measure first, optimize second. If you do not measure, you are guessing. Guessing is expensive—you might spend days rewriting a function that accounts for 0.1% of runtime while ignoring the function that accounts for 80%.

2.1 The Right Way to Measure Time

Python offers several ways to measure time. Use the right tool for the job:

  • time.time(): Returns wall-clock time (seconds since epoch). Do not use this for benchmarking. It can jump backward if the system clock adjusts.
  • time.perf_counter(): High-resolution timer for short intervals. Use this. It measures elapsed time with the highest available resolution.
  • time.process_time(): Measures CPU time spent by the current process only, excluding sleep. Good for understanding CPU burn vs. waiting.

2.2 The timeit Module

For microbenchmarks (small snippets), use the timeit module. It disables garbage collection, runs multiple trials, and reports the best time (which reduces noise from OS scheduling).

Using timeit
import timeit

# Measure list comprehension vs for-loop
setup = "data = list(range(1000))"

loop_code = """
result = []
for x in data:
    result.append(x * 2)
"""

comp_code = "result = [x * 2 for x in data]"

t_loop = timeit.timeit(loop_code, setup=setup, number=10000)
t_comp = timeit.timeit(comp_code, setup=setup, number=10000)

print(f"Loop:  {t_loop:.4f}s")
print(f"Comp:  {t_comp:.4f}s")
print(f"Speedup: {t_loop/t_comp:.1f}x")

2.3 Statistical Rigor in Benchmarking

Computers are noisy. Your benchmark is affected by:

  • OS Scheduling: The operating system pauses your program to run other tasks.
  • CPU Turbo Boost: Modern CPUs slow down when hot. The first run may be faster.
  • Cache Warm-up: The first access to data is slow (cache miss). Subsequent accesses are fast (cache hit).
  • Garbage Collection: Python may pause to clean up memory mid-benchmark.

Best practices:

  1. Run multiple trials (at least 7) and take the median, not the mean. Median is robust to outliers.
  2. Report the minimum, median, and standard deviation.
  3. Warm up the code with 1-2 throwaway runs before timing.
  4. Run on a quiet machine (close browsers and background apps).

2.4 A Robust Benchmarking Template

Here is a reusable function you should copy into every Colab notebook:

benchmark.py — Your new best friend
import time
import statistics

def benchmark(func, *args, n_trials=7, warmup=1, **kwargs):
    """
    Robust benchmarking function.
    Returns dict with min, median, mean, stdev times in milliseconds.
    """
    # Warmup runs
    for _ in range(warmup):
        func(*args, **kwargs)
    
    times = []
    for _ in range(n_trials):
        gc.collect()  # Clean slate
        start = time.perf_counter()
        result = func(*args, **kwargs)
        end = time.perf_counter()
        times.append((end - start) * 1000)  # ms
    
    return {
        'min': min(times),
        'median': statistics.median(times),
        'mean': statistics.mean(times),
        'stdev': statistics.stdev(times) if len(times) > 1 else 0,
        'result': result
    }

# Example usage:
def slow_sum(n):
    return sum(range(n))

stats = benchmark(slow_sum, 1_000_000)
print(f"Median: {stats['median']:.2f} ms ± {stats['stdev']:.2f} ms")

3. Big O & Algorithmic Complexity

Big O notation describes how much an algorithm's resource usage (time or memory) grows relative to its input size. It is the single most important concept in performance engineering because a better algorithm can turn an overnight job into a millisecond job.

3.1 The Intuition

Imagine you are looking for a specific book in a library:

  • O(1) — Constant Time: You know exactly where the book is. You walk directly to shelf 4, row 2. The time does not depend on how big the library is.
  • O(log n) — Logarithmic: The library is sorted by title. You open the catalog in the middle. If your title is alphabetically before the middle, you search the left half; otherwise, the right half. Each step halves the problem. A library with 1,000,000 books needs only ~20 checks.
  • O(n) — Linear: The library is unsorted. You scan every shelf one by one. Twice as many books takes twice as long.
  • O(n log n) — Linearithmic: You want to sort the books by title. An efficient sorting algorithm (like Merge Sort or Timsort) does this by repeatedly dividing and merging. It is only slightly worse than linear.
  • O(n²) — Quadratic: You want to find all pairs of books written by the same author. You compare every book with every other book. 10 books → 100 comparisons. 1,000 books → 1,000,000 comparisons.
  • O(2^n) — Exponential: You want to try every possible subset of books to fit in your bag. Each book is either "in" or "out." 10 books → 1,024 subsets. 50 books → 1 quadrillion subsets. Impossible.
Figure 3.1: Growth rates of common Big O classes. Notice how O(n²) explodes compared to O(n log n).

3.3 Amortized Analysis

Sometimes an individual operation is slow, but the average over many operations is fast. Python's list.append() is the classic example.

Appending to a Python list is usually O(1). But when the list runs out of pre-allocated space, Python must allocate a new, larger array and copy all elements over. That single append is O(n). However, because Python over-allocates (typically 1.125x growth), the average cost across millions of appends is still O(1). We call this amortized O(1).

3.4 Colab Experiment: Complexity in Action

Let's watch different complexities explode in real time.

Experiment 3.1: Timing Different Complexities
import time
import matplotlib.pyplot as plt

sizes = [100, 200, 400, 800, 1600, 3200]
times_constant = []
times_linear = []
times_quadratic = []

for n in sizes:
    data = list(range(n))
    
    # O(1): Access first element
    start = time.perf_counter()
    _ = data[0]
    times_constant.append((time.perf_counter()-start)*1e6)
    
    # O(n): Sum all elements
    start = time.perf_counter()
    _ = sum(data)
    times_linear.append((time.perf_counter()-start)*1e6)
    
    # O(n^2): Nested loop (avoid n > 5000!)
    start = time.perf_counter()
    total = 0
    for i in data:
        for j in data:
            total += i*j
    times_quadratic.append((time.perf_counter()-start)*1e6)

plt.figure(figsize=(10,5))
plt.plot(sizes, times_constant, 'o-', label='O(1) - constant')
plt.plot(sizes, times_linear, 's-', label='O(n) - linear')
plt.plot(sizes, times_quadratic, '^-', label='O(n²) - quadratic')
plt.xlabel('Input Size'); plt.ylabel('Time (microseconds)')
plt.legend(); plt.yscale('log'); plt.grid(True)
plt.title('Complexity Growth Visualization')
plt.show()

Observation: On a log scale, O(n) looks like a straight line with slope 1. O(n²) has slope 2. O(1) is flat. If your code's timing curve looks like O(n²) but you expected O(n), you have a nested loop bug.

4. CPU Profiling with cProfile

Profiling is like giving your code an MRI scan. Instead of guessing where the bottleneck is, you instrument the code to record exactly how much time is spent in each function. cProfile is Python's built-in profiler. It is deterministic (records every call) and has low enough overhead for most quant workflows.

4.1 How cProfile Works

cProfile hooks into the Python interpreter. Every time a function is called or returns, cProfile records the time. After the program finishes, it prints a table showing:

  • ncalls: Number of calls
  • tottime: Total time spent in the function, excluding subcalls. This tells you if the function itself is slow.
  • cumtime: Cumulative time spent in the function, including subcalls. This tells you if the function calls slow things.
  • percall: Average time per call

4.2 Basic Usage

Profiling a script
import cProfile
import pstats
import io

def heavy_computation(n):
    return sum(i*i for i in range(n))

def main():
    result = heavy_computation(1_000_000)
    print(f"Result: {result}")

# Profile it
profiler = cProfile.Profile()
profiler.enable()
main()
profiler.disable()

# Print sorted stats
s = io.StringIO()
ps = pstats.Stats(profiler, stream=s).sort_stats('cumulative')
ps.print_stats(10)
print(s.getvalue())

4.3 Reading the Output

The output table is dense. Here is how to read it:

   ncalls  tottime  percall  cumtime  percall filename:lineno(function)
        1    0.123    0.123    0.456    0.456 script.py:10(main)
  1000000    0.200    0.000    0.200    0.000 {built-in method builtins.sum}
        1    0.133    0.133    0.133    0.133 script.py:5(heavy_computation)

In this example:

  • main has high cumtime (0.456) but low tottime (0.123). It is not slow itself; it calls slow things.
  • sum (built-in) has tottime = cumtime = 0.200. This is where the actual work happens.
  • heavy_computation has equal tottime and cumtime (0.133), meaning it does work itself and calls nothing slow.

4.5 Colab Experiment: Find the Bottleneck

Here is a deliberately slow quant-style script. Your job is to profile it and identify the top 3 time sinks.

Experiment 4.1: The Slow Backtest
import random
import cProfile, pstats, io

def generate_prices(n):
    """Generate fake price series"""
    prices = [100.0]
    for _ in range(n):
        prices.append(prices[-1] * (1 + random.gauss(0, 0.02)))
    return prices

def moving_average(prices, window):
    """Naive moving average"""
    result = []
    for i in range(len(prices) - window):
        avg = sum(prices[i:i+window]) / window
        result.append(avg)
    return result

def generate_signals(prices, ma):
    """Generate buy/sell signals"""
    signals = []
    for i in range(len(ma)):
        if prices[i+window] > ma[i]:
            signals.append('BUY')
        else:
            signals.append('SELL')
    return signals

def run_backtest():
    prices = generate_prices(50000)
    ma = moving_average(prices, 20)
    signals = generate_signals(prices, ma)
    return len([s for s in signals if s == 'BUY'])

# Profile this
pr = cProfile.Profile()
pr.enable()
run_backtest()
pr.disable()

s = io.StringIO()
pstats.Stats(pr, stream=s).sort_stats('tottime').print_stats(15)
print(s.getvalue())

Questions to answer:

  1. Which function has the highest tottime?
  2. What percentage of total time is spent in random.gauss?
  3. Is the time dominated by computation or by Python interpreter overhead?

5. Line-by-Line Profiling

cProfile tells you which function is slow. line_profiler tells you which line inside that function is slow. This is crucial when a function contains 50 lines of mixed logic—some fast, some catastrophic.

5.1 Installation & Usage

line_profiler is not in the standard library. Install it:

!pip install line_profiler

Use the @profile decorator on the function you want to inspect. Then run with kernprof:

$ kernprof -l -v script.py

In Colab/Jupyter, use the magic command:

%load_ext line_profiler
%lprun -f my_function my_function(args)

5.2 Reading line_profiler Output

The output shows each line with:

  • Line #: Line number
  • Hits: Number of times executed
  • Time: Total time spent on that line
  • Per Hit: Average time per execution
  • % Time: Percentage of total function time
  • Line Contents: The actual code

Look for lines with high % Time. Often, the bottleneck is a single line inside a loop.

6. Memory Management & Profiling

Speed is not the only resource that matters. Memory is equally critical. A quant script that processes tick data can easily allocate gigabytes of RAM. If you run out of memory, your program crashes or starts swapping to disk, making it thousands of times slower. Worse, memory leaks in long-running trading infrastructure can cause outages.

6.1 How Python Manages Memory

Python uses two mechanisms:

  1. Reference Counting: Every Python object has a counter tracking how many variables point to it. When the count hits zero, the memory is freed immediately. This is simple and fast but cannot handle circular references (A points to B, B points to A).
  2. Generational Garbage Collection: A secondary collector detects circular references. It classifies objects into three generations (0, 1, 2). New objects start in generation 0. If they survive a collection, they move to generation 1, then 2. Older generations are scanned less frequently.
Figure 6.1: Reference Counting. The integer object 42 has refcount=2 because both x and y point to it.

6.3 Memory Optimization Techniques

  • Generators: Use yield instead of building massive lists. A generator produces one item at a time.
  • __slots__: Normally, Python objects store attributes in a dictionary (__dict__), which is memory-heavy. __slots__ pre-declares attributes, eliminating the dict.
  • Arrays vs Lists: A list of floats stores Python objects (28 bytes each + pointer). A numpy.array stores raw C doubles (8 bytes each).
  • Object Pools: Reuse objects instead of creating new ones in hot loops.
__slots__ example
class HeavyPoint:
    def __init__(self, x, y):
        self.x = x
        self.y = y

class LightPoint:
    __slots__ = ('x', 'y')
    def __init__(self, x, y):
        self.x = x
        self.y = y

# LightPoint uses ~50% less memory per instance

7. NumPy Optimization

NumPy is the bedrock of quant computing in Python. It provides N-dimensional arrays backed by highly optimized C and Fortran code. The key to NumPy performance is vectorization: replacing Python loops with array-wide operations that execute in compiled code.

7.1 Why Python Loops Are Slow

Every iteration of a Python for loop does the following:

  1. Fetch the iterator object
  2. Call __next__()
  3. Check the type of the element
  4. Look up the operation (e.g., +) in the object's method table
  5. Allocate a new Python object for the result
  6. Store the result

This overhead is ~50-200 nanoseconds per iteration. For 1 million iterations, that is 50-200 milliseconds of pure overhead—before doing any actual math. NumPy avoids all of this by operating on typed, contiguous memory blocks in C.

Figure 7.1: Vectorization replaces many scalar operations with one batched operation.

7.2 Broadcasting

Broadcasting allows NumPy to perform arithmetic on arrays of different shapes without explicit loops. The rules are:

  1. If arrays have different dimensions, prepend 1s to the smaller shape.
  2. Two dimensions are compatible if they are equal or one of them is 1.
Broadcasting examples
import numpy as np

a = np.array([1, 2, 3])          # shape (3,)
b = np.array([[10], [20], [30]])  # shape (3,1)

# Broadcasting: a becomes (3,3), b becomes (3,3)
print((a + b).shape)  # (3, 3)

7.3 Views vs Copies

NumPy can return a view (shares memory) or a copy (new memory). Views are instant; copies are expensive.

  • View: Slicing (arr[::2]), reshape, ravel
  • Copy: Fancy indexing (arr[[0,2,4]]), flatten, arr.T.copy()

Check with arr.flags['OWNDATA'] or np.shares_memory(a, b).

import numpy as np
import time

prices = np.random.randn(1_000_000)

# Method 1: Pure Python loop
def ma_loop(prices, window):
    result = []
    for i in range(len(prices)-window):
        result.append(sum(prices[i:i+window])/window)
    return np.array(result)

# Method 2: Cumulative sum vectorization
def ma_cumsum(prices, window):
    cumsum = np.cumsum(np.insert(prices, 0, 0))
    return (cumsum[window:] - cumsum[:-window]) / window

# Method 3: NumPy convolution
def ma_convolve(prices, window):
    return np.convolve(prices, np.ones(window)/window, mode='valid')

    print(f"{name:10}: {elapsed:8.2f} ms | shape: {r.shape}")

8. Pandas Optimization

Pandas is built on top of NumPy but adds labels, missing data handling, and heterogeneous types. This power comes with overhead. The difference between a well-written Pandas pipeline and a poorly written one can be 100x or more.

8.1 How Pandas Works Internally

A DataFrame is not just a Python dict of lists. Internally, Pandas uses a BlockManager that groups columns by data type into contiguous NumPy arrays (blocks). This means:

  • Operations on a single column are fast (one NumPy array).
  • Row-wise operations are slow (must access multiple blocks).
  • Adding/removing columns is cheap. Adding/removing rows is expensive.

9. Concurrency & Parallelism

Modern CPUs have multiple cores. A single Python process uses only one core at a time due to the Global Interpreter Lock (GIL). To use all cores, you must understand the concurrency landscape.

9.1 The GIL Explained

The GIL is a mutex (lock) that protects access to Python objects. It ensures that only one thread executes Python bytecode at a time. This means:

  • Threading does NOT speed up CPU-bound Python code. Threads alternate, not run simultaneously.
  • Multiprocessing spawns new Python processes, each with its own GIL and memory space. This bypasses the GIL but has overhead.
  • Asyncio uses a single thread with an event loop. Perfect for waiting on thousands of network connections.
Figure 9.1: The Global Interpreter Lock allows only one thread to hold the Python interpreter at a time.

10. JIT Compilation with Numba & Cython

When vectorization is impossible (e.g., complex loops with conditionals), Just-In-Time (JIT) compilation can bridge the gap. JIT compilers translate Python code to fast machine code at runtime.

10.1 Numba

Numba is the easiest JIT compiler for Python. Add @jit or @njit (no-Python mode) to a function, and Numba compiles it to LLVM bytecode.

Numba basics
from numba import njit, prange
import numpy as np

@njit
def sum_2d(arr):
    total = 0.0
    for i in range(arr.shape[0]):
        for j in range(arr.shape[1]):
            total += arr[i, j]
    return total

# First call compiles (slow), subsequent calls are fast
arr = np.random.rand(1000, 1000)
print(sum_2d(arr))

10.2 Numba Advanced Features

  • parallel=True: Automatically parallelize loops (releases GIL)
  • prange: Parallel range (like OpenMP)
  • fastmath=True: Allow unsafe floating-point optimizations for speed
  • cache=True: Cache compiled function to disk
Parallel Numba
@njit(parallel=True, fastmath=True)
def parallel_sum(arr):
    total = 0.0
    for i in prange(arr.shape[0]):
        for j in range(arr.shape[1]):
            total += arr[i, j]
    return total

10.3 When Numba Fails

Numba's @njit requires:

  • NumPy arrays and scalars only (no Pandas DataFrames)
  • No arbitrary Python objects or C extensions
  • No string manipulation (limited support)

If Numba cannot compile a function, it falls back to object mode (slow). Use @njit (not @jit) to force an error instead of silent fallback.

10.4 Cython Intro

Cython is a static compiler. You write .pyx files with type declarations, compile them to C, and import as Python modules. It is more powerful than Numba but has a steeper learning curve.

Cython example (conceptual)
# In a .pyx file:
cimport numpy as np
def cython_sum(double[:] arr):
    cdef double total = 0
    cdef int i
    for i in range(arr.shape[0]):
        total += arr[i]
    return total

11. I/O Optimization & Data Formats

You can have the fastest algorithm in the world, but if reading data takes 99% of your runtime, you have an I/O bottleneck. Quant workflows are often data-bound, not compute-bound.

11.1 The I/O Bottleneck

Reading a 1 GB CSV file can take 30-60 seconds in Pandas. The same data in Parquet format takes 2-5 seconds. Why?

  • CSV is text. Every number must be parsed from ASCII to binary. There is no type information.
  • Parquet is binary and columnar. It stores data in compressed, typed columns. You can read only the columns you need.
  • Feather is binary and optimized for fast I/O between Python and R.
  • HDF5 is hierarchical, good for random access to large datasets.

HDF5BinaryFastFastPartialLarge arrays, random access

11.2 Memory Mapping

Memory mapping (np.memmap) allows you to treat a file on disk as if it were in RAM. The OS loads pages on demand and evicts them when memory is tight. This lets you work with files larger than your physical RAM.

Memory-mapped array
# Create a memory-mapped array
fp = np.memmap('data.bin', dtype='float64', mode='w+', shape=(1000000, 100))
fp[:] = np.random.rand(1000000, 100)
del fp  # Flush to disk

# Later, read only a slice without loading everything
fp = np.memmap('data.bin', dtype='float64', mode='r', shape=(1000000, 100))
subset = fp[1000:2000, :]  # Only these pages are loaded from disk

11.3 Caching Strategies

If you compute the same thing repeatedly, cache it:

  • functools.lru_cache: Memoize function results in memory.
  • Disk caching: Use joblib.Memory or diskcache to persist results across runs.

12. Case Study: Optimizing a Quant Pipeline

This section ties everything together. We will take a realistic, slow quant pipeline and optimize it step-by-step using the techniques from this guide.

12.1 The Problem

You have a CSV of 1 million historical trades. You need to:

  1. Load the data
  2. Calculate a 20-period rolling VWAP per symbol
  3. Generate signals when price crosses above VWAP
  4. Backtest a simple strategy
  5. Run parameter sweep over 3 window sizes and 5 symbols

12.2 Baseline (The Slow Version)

baseline.py — Before optimization
import pandas as pd
import numpy as np
import time

def baseline_pipeline(file_path):
    # Step 1: Load
    df = pd.read_csv(file_path)
    
    # Step 2: Calculate VWAP per symbol using loop
    results = []
    for symbol in df['symbol'].unique():
        sub = df[df['symbol'] == symbol].copy()
        sub['vwap'] = 0.0
        for i in range(len(sub)):
            window = sub.iloc[max(0, i-19):i+1]
            sub.iloc[i, sub.columns.get_loc('vwap')] = (window['price'] * window['volume']).sum() / window['volume'].sum()
        results.append(sub)
    
    # Step 3: Concatenate
    df = pd.concat(results)
    
    # Step 4: Signal
    df['signal'] = 0
    for i in range(1, len(df)):
        if df.iloc[i]['price'] > df.iloc[i]['vwap']:
            df.iloc[i, df.columns.get_loc('signal')] = 1
    
    return df

# This would take MINUTES to run on 1M rows

12.3 Optimization Walkthrough

Step 1: Profile — Run cProfile. The nested loops dominate.

Step 2: Algorithm — VWAP can be computed with cumulative sums, not nested loops. O(n²) → O(n).

Step 3: Vectorize — Use groupby + cumsum for VWAP. Use vectorized comparison for signals.

Step 4: Format — Convert CSV to Parquet for faster loading.

Step 5: Parallelize — Run parameter sweep with ProcessPoolExecutor.

Step 6: Memory — Use categorical for symbol column. Delete intermediate DataFrames.

optimized.py — After optimization
import pandas as pd
import numpy as np
from concurrent.futures import ProcessPoolExecutor

def fast_vwap(df, window=20):
    df = df.copy()
    df['cum_pv'] = (df['price'] * df['volume']).groupby(df['symbol']).cumsum()
    df['cum_vol'] = df['volume'].groupby(df['symbol']).cumsum()
    
    # Subtract rolling window start using shift
    df['pv_win'] = df['cum_pv'] - df.groupby('symbol')['cum_pv'].shift(window, fill_value=0)
    df['vol_win'] = df['cum_vol'] - df.groupby('symbol')['cum_vol'].shift(window, fill_value=0)
    df['vwap'] = df['pv_win'] / df['vol_win']
    
    df['signal'] = (df['price'] > df['vwap']).astype(int)
    return df

def run_sweep(param_list):
    with ProcessPoolExecutor() as ex:
        results = list(ex.map(fast_vwap, param_list))
    return results
