2. Data Formats: JSON, CSV, Parquet

Data APIs and ETL pipelines shuffle data between systems. These are the containers that data travels in.

JSON (JavaScript Object Notation)

The universal API language. Human-readable, nested, and supported everywhere.

JSON
{
  "ticker": "AAPL",
  "price": 150.75,
  "metadata": {
    "exchange": "NASDAQ",
    "currency": "USD"
  },
  "trades": [
    {"time": "09:30:00", "price": 150.20, "size": 100},
    {"time": "09:30:01", "price": 150.75, "size": 250}
  ]
}

In Python: json.loads() converts JSON string → Python dict. json.dumps() does the reverse.

CSV (Comma-Separated Values)

The simplest tabular format. Great for human inspection, poor for nested data.

CSV
date,ticker,open,high,low,close,volume
2024-01-15,AAPL,150.00,152.50,149.80,151.20,45000000
2024-01-16,AAPL,151.20,153.00,150.50,152.80,38000000

Parquet

The columnar storage champion. Compressed, schema-enforced, and lightning-fast for analytics. In quant workflows, historical tick data is often stored in Parquet.

Rule of Thumb: API responses come as JSON. You analyze with Pandas. You archive with Parquet.

4. SQL & Database Basics

SELECT date, close_price, volume

🔌 APIs: The Data Lifeline

APIs (Application Programming Interfaces) are how your code talks to the world. For a quant engineer, APIs are the primary mechanism for ingesting market data, fundamentals, alternative data, and executing trades.

What is an API? (The Restaurant Analogy)

Imagine a restaurant kitchen. You can't walk in and cook your own meal. Instead, you:

  1. Look at the menu (API Documentation)
  2. Place an order (Send a Request)
  3. The kitchen prepares it (Server processes)
  4. A waiter brings your food (Response returned)

The API is the menu + waiter + kitchen protocol. It defines what you can ask for and how to ask for it.

API Architecture Flow

REST API Anatomy

A REST API request has four key components. Think of them as the address, method, ID card, and envelope of your data request.

1. Endpoint (The Address)

URL Structure
https://api.example.com/v1/market-data/price?symbol=AAPL&date=2024-01-15
|------- Base URL -------|--Version--|--Resource--|--Path--|---Query Parameters---|

2. HTTP Method (The Action)

We covered these in Prerequisites, but in API context:

  • GET /prices → Retrieve all prices
  • GET /prices/AAPL → Retrieve AAPL price
  • POST /prices → Add a new price record
  • PUT /prices/AAPL → Update AAPL price entirely
  • PATCH /prices/AAPL → Partial update

3. Headers (The Metadata)

HTTP Headers
Content-Type: application/json        # What we're sending
Accept: application/json              # What we want back
Authorization: Bearer sk_live_abc123  # Who we are
X-RateLimit-Limit: 100                # Max requests allowed

4. Body (The Payload)

Used with POST/PUT. Contains the actual data.

JSON Body
{
  "ticker": "AAPL",
  "price": 151.20,
  "timestamp": "2024-01-15T09:30:00Z"
}

Python Requests in Depth

The requests library is your primary tool. Here's how to use it professionally.

Python
import requests
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

# Professional session setup with retries
session = requests.Session()
retries = Retry(
    total=5,
    backoff_factor=1,      # Wait 1s, 2s, 4s, 8s...
    status_forcelist=[429, 500, 502, 503, 504]
)
session.mount('https://', HTTPAdapter(max_retries=retries))

# GET request with parameters
url = "https://api.example.com/v1/prices"
params = {
    "symbol": "AAPL",
    "start_date": "2024-01-01",
    "end_date": "2024-01-31"
}
headers = {
    "Authorization": "Bearer YOUR_API_KEY",
    "Accept": "application/json"
}

response = session.get(url, params=params, headers=headers, timeout=10)

# Always check status before parsing
if response.status_code == 200:
    data = response.json()  # Convert JSON to Python dict/list
    print(f"Retrieved {len(data)} records")
elif response.status_code == 429:
    print("Rate limited. Backing off...")
else:
    print(f"Error {response.status_code}: {response.text}")

Pagination: When Data Doesn't Fit in One Call

Asking for "all historical prices since 1990" in one request will crash or timeout. APIs paginate — they return data in chunks.

Pagination Strategies
Offset Pagination
Page 1: offset=0, limit=100
Page 2: offset=100, limit=100
Page 3: offset=200, limit=100
Simple but slow at deep pages
Cursor Pagination
Page 1: cursor=abc123
Page 2: cursor=def456 (from prev)
Page 3: cursor=ghi789 (from prev)
Fast & consistent, preferred
Python: Offset Pagination Loop
def fetch_all_prices(symbol, api_key):
    all_data = []
    page = 1
    while True:
        response = requests.get(
            "https://api.example.com/prices",
            params={"symbol": symbol, "page": page, "per_page": 100},
            headers={"Authorization": f"Bearer {api_key}"}
        )
        batch = response.json()
        if not batch:
            break
        all_data.extend(batch)
        page += 1
    return all_data

Rate Limiting: The API Speed Bump

APIs cost money to run. Providers limit how often you can call them. Violating limits gets you blocked (HTTP 429).

Strategies to Handle Rate Limits

Python
import time

def rate_limited_request(url, params, headers, max_per_minute=5):
    """Simple rate limiter using sleep."""
    time.sleep(60 / max_per_minute)  # Sleep before request
    return requests.get(url, params=params, headers=headers)

# Better: Token bucket or using a library
from ratelimit import limits, sleep_and_retry

@sleep_and_retry
@limits(calls=5, period=60)  # 5 calls per minute
def call_api(url, params, headers):
    return requests.get(url, params=params, headers=headers)
💡 Pro Tip: Always check X-RateLimit-Remaining and X-RateLimit-Reset headers in responses. Sleep proactively, not just after getting 429.

API Extraction Patterns

Pattern 1: Full Extraction

Pull everything, every time. Simple but slow. Best for small reference data (company info, sector mappings).

Python
def extract_full(url, headers):
    """Extract entire dataset. Use only for small tables."""
    response = requests.get(url, headers=headers, timeout=30)
    response.raise_for_status()
    return response.json()

Pattern 2: Incremental Extraction

Pull only what changed since last run. Uses a "watermark" (timestamp or ID). Essential for large datasets like daily prices.

Python
def extract_incremental(url, headers, last_watermark):
    """
    Extract only records updated since last_watermark.
    last_watermark is stored from previous run (e.g., in a file or DB).
    """
    params = {"updated_after": last_watermark}
    response = requests.get(url, params=params, headers=headers)
    data = response.json()
    
    # Update watermark for next run
    new_watermark = max(record["updated_at"] for record in data)
    save_watermark(new_watermark)
    
    return data

Pattern 3: Change Data Capture (CDC)

Database-specific. Uses transaction logs or triggers to capture changes. Common in enterprise quant systems.

Quant Example: Your portfolio management system has a PostgreSQL database. CDC captures every INSERT/UPDATE to the trades table and streams it to your analytics warehouse in real-time.

Robust Extraction: The Professional Template

This is a battle-tested extraction function you can adapt for any API source.

Python: production_extractor.py
import requests
import logging
import time
from datetime import datetime
from typing import Dict, List, Optional

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

class APIExtractor:
    def __init__(self, base_url: str, api_key: str, rate_limit_per_sec: float = 5.0):
        self.base_url = base_url
        self.session = requests.Session()
        self.session.headers.update({
            "Authorization": f"Bearer {api_key}",
            "Content-Type": "application/json"
        })
        self.min_interval = 1.0 / rate_limit_per_sec
        self.last_request_time = 0
        
    def _rate_limit(self):
        """Ensure we don't exceed rate limits."""
        elapsed = time.time() - self.last_request_time
        if elapsed < self.min_interval:
            time.sleep(self.min_interval - elapsed)
        self.last_request_time = time.time()
    
    def extract(self, endpoint: str, params: Optional[Dict] = None) -> Dict:
        """
        Extract data with automatic retries, rate limiting, and logging.
        """
        url = f"{self.base_url}/{endpoint}"
        self._rate_limit()
        
        try:
            response = self.session.get(url, params=params, timeout=30)
            
            # Log metadata
            logger.info(f"Extracted {url} | Status: {response.status_code} | "
                       f"Remaining: {response.headers.get('X-RateLimit-Remaining', 'N/A')}")
            
            response.raise_for_status()
            return {
                "data": response.json(),
                "extracted_at": datetime.utcnow().isoformat(),
                "status_code": response.status_code,
                "url": url
            }
            
        except requests.exceptions.RequestException as e:
            logger.error(f"Extraction failed for {url}: {e}")
            raise

# Usage
extractor = APIExtractor("https://api.polygon.io", "YOUR_KEY", rate_limit_per_sec=5)
result = extractor.extract("v2/aggs/ticker/AAPL/range/1/day/2024-01-01/2024-01-31")

Data Validation at Extract Time

Validate early. Catch schema changes, nulls, and type mismatches before they corrupt your warehouse.

Python
import pandas as pd
from typing import Dict, Any

class ExtractValidator:
    def __init__(self, schema: Dict[str, str]):
        """
        schema: {"column_name": "dtype", ...}
        e.g., {"ticker": "string", "price": "float64", "date": "datetime64[ns]"}
        """
        self.schema = schema
    
    def validate(self, df: pd.DataFrame) -> pd.DataFrame:
        # Check required columns
        missing = set(self.schema.keys()) - set(df.columns)
        if missing:
            raise ValueError(f"Missing columns: {missing}")
        
        # Check types
        for col, dtype in self.schema.items():
            if not pd.api.types.is_dtype_equal(df[col].dtype, dtype):
                try:
                    df[col] = df[col].astype(dtype)
                except Exception as e:
                    raise TypeError(f"Column {col} cannot cast to {dtype}: {e}")
        
        # Check for nulls in critical columns
        null_counts = df[list(self.schema.keys())].isnull().sum()
        if null_counts.any():
            print(f"⚠️ Nulls detected:\n{null_counts[null_counts > 0]}")
        
        return df

# Usage
validator = ExtractValidator({
    "ticker": "string",
    "date": "datetime64[ns]",
    "close": "float64",
    "volume": "Int64"
})
clean_df = validator.validate(raw_df)

1. Data Cleaning

Handling Missing Values

Missing data in financial time series is common (weekends, holidays, delisted stocks). You have three strategies:

  • Drop: Remove rows with nulls. Safe for large datasets, dangerous for time series.
  • Forward Fill: Use last known value. Standard for prices (Friday's close carries to Monday).
  • Interpolation: Estimate missing values. Useful for sparse fundamental data.
Python
import pandas as pd
import numpy as np

# Sample with missing values
df = pd.DataFrame({
    'date': pd.date_range('2024-01-01', periods=5, freq='D'),
    'price': [150.0, np.nan, np.nan, 153.0, 154.0]
})

# Strategy 1: Forward fill (ffill) — standard for prices
df['price_ffill'] = df['price'].ffill()

# Strategy 2: Linear interpolation
df['price_interp'] = df['price'].interpolate(method='linear')

# Strategy 3: Fill with specific value (e.g., 0 for volume)
df['volume'] = [1000, 2000, np.nan, 1500, 1800]
df['volume_filled'] = df['volume'].fillna(0)

print(df)

Outlier Detection

A stock price jumping 50% in one minute is either a corporate action or bad data. You must detect and handle outliers.

Python
# Z-score method: flag values > 3 standard deviations
from scipy import stats
df['z_score'] = np.abs(stats.zscore(df['price'].dropna()))
df['is_outlier'] = df['z_score'] > 3

# IQR method: more robust to extreme values
Q1 = df['price'].quantile(0.25)
Q3 = df['price'].quantile(0.75)
IQR = Q3 - Q1
df['is_outlier_iqr'] = (df['price'] < (Q1 - 1.5 * IQR)) | (df['price'] > (Q3 + 1.5 * IQR))

3. Structuring: Normalization vs Denormalization

Normalization (Database Form)

Split data into related tables to avoid redundancy. Good for storage, bad for analysis.

Conceptual Schema
Table: companies
  ticker (PK) | name       | sector     | market_cap
  AAPL        | Apple Inc. | Technology | 3000000000000

Table: prices
  id (PK) | ticker (FK) | date       | close | volume
  1       | AAPL        | 2024-01-15 | 151.2 | 45000000

Join everything into one flat table. Redundant but fast to query. This is what quants usually want.

4. Feature Engineering for Quants

This is where raw prices become signals. These transformations are the bread and butter of quantitative analysis.

Returns

Python
# Simple returns: (P_t / P_{t-1}) - 1
df['simple_return'] = df['close'].pct_change()

# Log returns: ln(P_t / P_{t-1})
df['log_return'] = np.log(df['close'] / df['close'].shift(1))

# Cumulative returns
df['cum_return'] = (1 + df['simple_return']).cumprod() - 1

Moving Averages & Rolling Statistics

Python
# Simple Moving Average (SMA)
df['sma_20'] = df['close'].rolling(window=20).mean()

# Exponential Moving Average (EMA)
df['ema_20'] = df['close'].ewm(span=20, adjust=False).mean()

# Rolling volatility (standard deviation of returns)
df['volatility_20'] = df['log_return'].rolling(window=20).std() * np.sqrt(252)

# Rolling max/min
df['rolling_high_20'] = df['high'].rolling(window=20).max()

Technical Indicators (RSI)

Python
# Relative Strength Index (RSI)
delta = df['close'].diff()
gain = (delta.where(delta > 0, 0)).rolling(window=14).mean()
loss = (-delta.where(delta < 0, 0)).rolling(window=14).mean()
rs = gain / loss
df['rsi_14'] = 100 - (100 / (1 + rs))
💡 Quant Tip: Always use .shift(1) to ensure you're not using future data. A common bug is calculating returns without shifting, which creates look-ahead bias — your backtest will look amazing and fail in production.

5. Window Functions & Grouped Operations

Window functions let you calculate metrics "relative to" each row — e.g., "rank this stock's return within its sector."

Python
# Rank stocks by daily return within their sector
df['return_rank'] = df.groupby('date')['simple_return'].rank(ascending=False)

# Percentile rank (0 to 1)
df['return_pctile'] = df.groupby('date')['simple_return'].rank(pct=True)

# Cumulative sum within group
df['cumulative_volume'] = df.groupby('ticker')['volume'].cumsum()

# Expanding window (all history up to current row)
df['expanding_mean'] = df.groupby('ticker')['close'].expanding().mean().reset_index(0, drop=True)

6. Validation in Transform

After transforming, verify your assumptions before loading.

Python
def validate_transformed(df: pd.DataFrame) -> bool:
    checks = []
    
    # 1. No nulls in critical columns
    checks.append(df[['ticker', 'date', 'close']].isnull().sum().sum() == 0)
    
    # 2. Prices are positive
    checks.append((df['close'] > 0).all())
    
    # 3. Dates are within expected range
    checks.append(df['date'].max() <= pd.Timestamp.now())
    
    # 4. No duplicate ticker-date pairs
    checks.append(df.duplicated(subset=['ticker', 'date']).sum() == 0)
    
    # 5. Returns are reasonable (not ±1000%)
    checks.append(df['simple_return'].abs().max() < 10)
    
    if all(checks):
        print("✅ All validation checks passed")
        return True
    else:
        failed = [i for i, c in enumerate(checks) if not c]
        raise ValueError(f"Validation failed for checks: {failed}")

📤 The Load Phase

Loading is the final handoff. Your clean, transformed data must land in a storage system where analysts, portfolio managers, and trading algorithms can access it reliably and quickly.

Loading Targets: Where Data Lives

Data Storage Hierarchy

Loading Patterns

Full Load (Truncate + Insert)

Delete all existing data, insert fresh data. Simple but disruptive. Use only for small reference tables.

SQL
BEGIN TRANSACTION;
    TRUNCATE TABLE daily_prices;
    INSERT INTO daily_prices SELECT * FROM staging_daily_prices;
COMMIT;

Incremental Load (Append New Data)

Only insert new records. Fast and safe. Requires a reliable watermark (date, ID, timestamp).

SQL
INSERT INTO daily_prices
SELECT * FROM staging_daily_prices s
WHERE s.date > (SELECT MAX(date) FROM daily_prices);

Upsert (Merge): Insert or Update

If record exists, update it. If not, insert it. Critical for corrections and late-arriving data.

PostgreSQL
INSERT INTO daily_prices (ticker, date, close, volume)
VALUES ('AAPL', '2024-01-15', 151.25, 45000000)
ON CONFLICT (ticker, date) 
DO UPDATE SET 
    close = EXCLUDED.close,
    volume = EXCLUDED.volume,
    updated_at = NOW();

Loading to Databases with Pandas

Pandas can write directly to SQL, but for production pipelines, you need more control.

Python: Safe Bulk Loading
from sqlalchemy import create_engine, text
import pandas as pd

engine = create_engine("postgresql://user:pass@localhost:5432/quantdb")

def load_to_warehouse(df: pd.DataFrame, table_name: str, if_exists: str = 'append'):
    """
    Load DataFrame to warehouse with transaction safety.
    if_exists: 'fail', 'replace', 'append'
    """
    with engine.begin() as connection:  # Auto-commit or rollback
        df.to_sql(
            name=table_name,
            con=connection,
            if_exists=if_exists,
            index=False,
            method='multi',       # Batch inserts
            chunksize=1000      # Insert 1000 rows at a time
        )
        print(f"Loaded {len(df)} rows to {table_name}")

# Usage
load_to_warehouse(transformed_df, "fact_daily_prices", if_exists="append")

Error Handling & Resilience

Idempotency

A task is idempotent if running it twice produces the same result as running it once. This is critical for retries.

Example: An upsert SQL query is idempotent. A simple INSERT without conflict handling is not — running it twice creates duplicates.

Retry Strategies

Python

@retry(
    stop=stop_after_attempt(5),

def fetch_with_backoff(url):
    """Retry with exponential backoff: 4s, 8s, 16s, 32s, 60s"""

Dead Letter Queues (DLQ)

When a record fails processing after all retries, send it to a DLQ instead of crashing the pipeline. Review DLQs manually.

Python
def process_record(record):
    try:
        transform(record)
        load(record)
    except Exception as e:
        logger.error(f"Failed record: {record}, Error: {e}")

Monitoring & Alerting

You cannot improve what you don't measure. Production pipelines need observability.

  • SLA (Service Level Agreement): "Pipeline must finish by 7 PM ET." Airflow can alert if breached.
  • Data Freshness: Alert if the latest record in the warehouse is > 24 hours old.
  • Volume Anomalies: Alert if today's row count is ±30% from average (possible source issue).
  • Schema Drift: Alert if new columns appear or expected columns disappear.

import time
from datetime import datetime
from typing import Callable, List, Dict

class MiniPipeline:
    def __init__(self, name: str):
        self.name = name
        self.tasks: Dict[str, Callable] = {}
        self.dependencies: Dict[str, List[str]] = {}
        self.results: Dict[str, any] = {}
    
    def add_task(self, name: str, func: Callable, depends_on: List[str] = None):
        self.tasks[name] = func
        self.dependencies[name] = depends_on or []
    
    def run(self):
        print(f"🚀 Starting pipeline: {self.name}")
        completed = set()
        
        while len(completed) < len(self.tasks):
            ready = [
                name for name in self.tasks 
                if name not in completed 
                and all(dep in completed for dep in self.dependencies[name])
            ]
            
            for name in ready:
                print(f"  ▶️ Running: {name}")
                try:
                    self.results[name] = self.tasks[name](self.results)
                    completed.add(name)
                    print(f"  ✅ Completed: {name}")
                except Exception as e:
                    print(f"  ❌ Failed: {name} — {e}")
                    raise
            
            if not ready and len(completed) < len(self.tasks):
                raise ValueError("Circular dependency detected!")
        
        print(f"🏁 Pipeline complete!")
        return self.results

# Define tasks
def extract_prices(results):
    time.sleep(1)
    return {"prices": [150, 151, 152], "ticker": "AAPL"}

def extract_fundamentals(results):
    time.sleep(1)
    return {"pe_ratio": 25.5, "sector": "Technology"}

def transform(results):
    time.sleep(0.5)
    prices = results["extract_prices"]["prices"]
    pe = results["extract_fundamentals"]["pe_ratio"]
    return {"avg_price": sum(prices)/len(prices), "pe": pe}

def load(results):
    time.sleep(0.5)
    print(f"    Loading: avg_price={results['transform']['avg_price']}, pe={results['transform']['pe']}")
    return "loaded"

# Build and run
pipeline = MiniPipeline("daily_quant_etl")
pipeline.add_task("extract_prices", extract_prices)
pipeline.add_task("extract_fundamentals", extract_fundamentals)
pipeline.add_task("transform", transform, depends_on=["extract_prices", "extract_fundamentals"])
pipeline.add_task("load", load, depends_on=["transform"])

pipeline.run()

The Two Deadly Sins of Quant Data

1. Look-Ahead Bias

Using information that wasn't available at the time of the decision. This makes backtests unrealistically profitable.

Example: You calculate today's return using today's close, then use that return as a feature to predict whether to buy at today's open. Impossible. You don't know the close until 4 PM.
Python: The Wrong Way
# WRONG: Using today's close to predict today's action
df['return'] = df['close'].pct_change()
df['signal'] = df['return'] > 0  # Uses future info!
Python: The Right Way
# CORRECT: Use yesterday's close for today's decision
df['return'] = df['close'].pct_change()
df['signal'] = df['return'].shift(1) > 0  # Lag by 1 period

2. Survivorship Bias

Only analyzing stocks that still exist. Delisted companies (Enron, Lehman Brothers) are excluded, inflating historical performance.

Example: You backtest a strategy on current S&P 500 constituents over the last 20 years. But the current S&P 500 companies are the winners. The losers were removed. Your strategy looks amazing because it only "traded" winners.
Solution: Point-in-Time Universe
# Maintain a historical membership table
membership = pd.read_sql("""
    SELECT ticker, date, is_member 
    FROM sp500_membership 
    WHERE date = '{as_of_date}'
""", engine)

# Only trade stocks that were in the index AT THAT TIME
tradable = prices.merge(membership[membership['is_member']], on=['ticker', 'date'])

Corporate Actions & Adjustments

Stock splits and dividends change the price but not the value. Raw prices are misleading for historical analysis.

Split Example

AAPL did a 4:1 split in Aug 2020. The price dropped from ~$500 to ~$125. Without adjustment, it looks like a 75% crash.

Dividend Adjustment

When a $1 dividend is paid, the stock drops by $1 at open. Adjusted prices subtract dividends to show true total return.

Python: Using Adjusted Close
import yfinance as yf

# Always use adjusted prices for backtests
df = yf.Ticker("AAPL").history(period="5y")

# 'Close' = raw price (unadjusted)
# 'Adj Close' = adjusted for splits and dividends
df['return'] = df['Adj Close'].pct_change()  # Correct for total return
Important: Adjusted prices are backward-adjusted. A split today changes all historical prices. If you save prices daily, you must re-adjust historical data after every corporate action.

Data Quality for Quantitative Analysis

Bad data is worse than no data. It gives you false confidence.

Automated Quality Checks

Python
def quant_data_quality_checks(df: pd.DataFrame) -> Dict[str, bool]:
    results = {}
    
    # 1. No gaps in trading days (for daily data)
    trading_days = pd.bdate_range(df['date'].min(), df['date'].max())
    missing_days = set(trading_days) - set(df['date'])
    results['no_gaps'] = len(missing_days) <= 3  # Allow some holidays
    
    # 2. Price consistency: high >= low, close in range
    results['ohl_consistency'] = (
        (df['high'] >= df['low']).all() and
        (df['close'] <= df['high']).all() and
        (df['close'] >= df['low']).all()
    )
    
    # 3. Volume is positive
    results['positive_volume'] = (df['volume'] > 0).all()
    
    # 4. No duplicate timestamps
    results['unique_timestamps'] = df['date'].is_unique
    
    # 5. Returns within reasonable bounds (±50% daily is suspicious)
    returns = df['close'].pct_change().abs()
    results['reasonable_returns'] = returns.max() < 0.50
    
    # 6. Price > 0 (stocks can't have zero/negative price)
    results['positive_prices'] = (df[['open','high','low','close']] > 0).all().all()
    
    return results

# The signal should be NaN for the first month (no history)
print(f"\nNaN signals (expected): {portfolio['momentum_signal'].isna().sum()}")
print(f"Look-ahead check passed: {(portfolio.groupby('year_month')['rank'].nunique() <= 5).all()}")
