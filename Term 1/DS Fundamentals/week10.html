1. What is a Trading Strategy?

🍕 Simple Analogy

A trading strategy is like a recipe. It tells you: "When these specific ingredients (market conditions) appear, do this action (buy or sell), and when something else happens, do the opposite." A good recipe works reliably. A bad recipe only worked once.

In quantitative finance, a trading strategy is a set of precise, mathematical rules that decide:

📈
Entry Signal
When to BUY an asset. Example: "Buy when the 10-day average price crosses above the 50-day average."
📉
Exit Signal
When to SELL. Example: "Sell when the price drops 5% below where we bought it."
⚖️
Position Sizing
How much to buy. Example: "Never put more than 10% of your capital in one trade."
🛑
Risk Management
Limits to protect capital. Example: "Stop all trading if total losses exceed 20%."

2. Understanding Time Series Data

Financial data is a time series — a sequence of values recorded over time. Each row in your data represents one moment in time.

OHLCV Candlestick Data — What Each Column Means
⚠️ Key Insight

The CLOSE price is only known AFTER the day ends. This seems obvious, but many beginners accidentally use today's close price to make a trading decision that was supposed to happen at the start of the day. That's look-ahead bias — and it will ruin your backtest.

3. Returns: How Profit & Loss Is Measured

A return measures how much an investment grew or shrank over a period. There are two types you'll always use:

Simple Return
Return = (Price_End - Price_Start) / Price_Start × 100%
Example: Buy at $100, sell at $110 → Return = (110-100)/100 = 10%
Log Return (used in quant finance)
Log Return = ln(Price_End / Price_Start)
Log returns are additive across time — if daily log return = 0.01, then 252-day return ≈ 252 × 0.01 = 2.52 (252%). Simpler to work with mathematically.
PYTHON 
import pandas as pd
import numpy as np

# Sample price series (closing prices)
prices = pd.Series([100, 102, 101, 105, 103, 108])

# Simple daily returns
simple_returns = prices.pct_change()
# Output: [NaN, 0.02, -0.0098, 0.0396, -0.019, 0.0485]

# Log returns (preferred in quant finance)
log_returns = np.log(prices / prices.shift(1))

# Cumulative return (total growth from start)
cumulative = (1 + simple_returns).cumprod() - 1
print(f"Total return: {cumulative.iloc[-1]:.2%}")  # → 8.00%

Sharpe = (Mean Annual Return − Risk-Free Rate) / Annual Std Deviation

Core Concept

Backtesting: Testing Strategies on the Past

Backtesting is the process of simulating how a trading strategy would have performed if you had actually traded it historically. Instead of risking real money to test an idea, you "replay" the past with your strategy turned on, and measure the results.

🎮 Video Game Analogy

Backtesting is like playing a sports video game where you simulate historical matches. You set your team's tactics (the strategy), then the simulation replays past games and tells you your score. But if the game cheat-codes and shows you future moves before you decide — your win rate is meaningless. This is exactly what look-ahead bias does.

The Backtesting Simulation Loop

At its core, every backtest is just a loop through time, making decisions at each step using only information available at that moment:

The Core Backtesting Loop

Your First Real Backtest: Moving Average Crossover

The simplest and most classic strategy. Buy when the fast (short) moving average crosses above the slow (long) moving average. Sell when it crosses below.

📊 What is a Moving Average?

A moving average smooths out price noise. The "20-day moving average" on a given day is simply the average of the last 20 closing prices. It tells you the trend. A faster (shorter) MA reacts quickly to price changes; a slower one shows the longer trend.

Moving Average Crossover Signal
PYTHON — MINIMAL BACKTEST 
import yfinance as yf
import pandas as pd
import numpy as np
import matplotlib.pyplot as plt

# ── 1. LOAD DATA ──────────────────────────────────────────────
df = yf.download('SPY', start='2015-01-01', end='2023-12-31')
df = df[['Close']].copy()

# ── 2. COMPUTE SIGNALS ────────────────────────────────────────
df['MA_fast'] = df['Close'].rolling(10).mean()  # 10-day MA
df['MA_slow'] = df['Close'].rolling(50).mean()  # 50-day MA

# Signal: 1 = long (hold), 0 = flat (no position)
df['signal'] = np.where(df['MA_fast'] > df['MA_slow'], 1, 0)

# IMPORTANT: Shift signal by 1 day!
# We can only ACT on tomorrow's open using today's signal.
# This single line prevents a common form of look-ahead bias!
df['position'] = df['signal'].shift(1)

# ── 3. COMPUTE RETURNS ────────────────────────────────────────
df['market_return'] = df['Close'].pct_change()
df['strat_return']  = df['position'] * df['market_return']

# ── 4. PERFORMANCE METRICS ────────────────────────────────────
total_ret  = (1 + df['strat_return']).cumprod().iloc[-1] - 1
ann_sharpe = df['strat_return'].mean() / df['strat_return'].std() * 252**0.5

print(f"Total Return: {total_ret:.2%}")
print(f"Annualized Sharpe: {ann_sharpe:.2f}")

# ── 5. PLOT ───────────────────────────────────────────────────
fig, ax = plt.subplots(figsize=(14, 5))
(1 + df['strat_return']).cumprod().plot(ax=ax, label='Strategy', color='cyan')
(1 + df['market_return']).cumprod().plot(ax=ax, label='Buy & Hold', color='orange', alpha=0.6)
ax.legend()
ax.set_title('MA Crossover Backtest vs Buy-and-Hold')
plt.show()
✅ Notice the .shift(1) Line

That single df['position'] = df['signal'].shift(1) is doing critical work. Without it, we'd be using today's signal to trade today — but the signal is computed using today's closing price which we'd only know after the market closes. The shift ensures we only trade on information from the previous day. This is our first defense against look-ahead bias!

Critical Error #1

Look-Ahead Bias: Using the Future to Predict Itself

Look-ahead bias occurs when a backtest accidentally uses information that would NOT have been available at the time of a trading decision. It's the #1 reason why backtests look amazing but strategies fail in live trading.

🏇 Horse Racing Analogy

Imagine you're betting on horse races. Look-ahead bias is like looking at the race results first, then going back and "predicting" the winner. Of course you'd be 100% accurate! But that's not a prediction — it's cheating. A real prediction must be made BEFORE the race happens, using only information available at that moment.

⚠️ The Devastating Effect

A strategy with look-ahead bias might show a Sharpe Ratio of 3.0 (extraordinary!) in backtesting. In live trading, the same strategy might produce a Sharpe of -0.5 (losing money). The gap isn't because markets changed — it's because the backtest was cheating.

The 5 Most Common Types of Look-Ahead Bias

01
Signal-Execution Lag (The .shift() Problem)
You compute a signal using today's closing price, then trade at today's closing price — which is impossible! By the time you know the close, the market is already closed. Fix: always shift signals by 1 period before computing returns.
02
Data Snooping / Curve Fitting
You test 1,000 parameter combinations on your full dataset, pick the best one (say, MA(23,47)), and report its results. But those parameters were chosen because they fit the past perfectly — they won't generalize. Fix: use a separate out-of-sample test set you never touched during optimization.
03
Survivorship Bias
You backtest a strategy on "the S&P 500 components" — but you use TODAY's list of companies. Companies that went bankrupt years ago aren't in that list. Your strategy has unknowingly selected only survivors, making past results look better than they were. Fix: use a point-in-time index membership database.
04
Using Future-Adjusted Prices Without Care
Most data providers give "split-adjusted" historical prices — they retroactively adjust all past prices for stock splits and dividends. If you use these to compute signals without care, you're using information about future corporate actions. Fix: use a "point-in-time" or "raw" price for signal generation, only use adjusted prices for return calculations.
05
Normalizing Entire Dataset Before Splitting
Many ML practitioners normalize data (e.g., fit a StandardScaler on ALL data including test set), then split. This leaks statistics from the future into the past. Fix: always fit your scaler/normalizer ONLY on training data, then apply to test data.

Code Example: Look-Ahead Bias in Action

PYTHON — BIAS DEMONSTRATION 
import yfinance as yf
import pandas as pd
import numpy as np

df = yf.download('AAPL', start='2015-01-01', end='2023-12-31')
df['MA10'] = df['Close'].rolling(10).mean()
df['MA50'] = df['Close'].rolling(50).mean()
df['daily_return'] = df['Close'].pct_change()

# ❌ BIASED VERSION — uses same-day signal to trade same day
df['biased_signal'] = np.where(df['MA10'] > df['MA50'], 1, 0)
biased_return = (df['biased_signal'] * df['daily_return']).sum()

# ✅ CORRECT VERSION — signal from yesterday, execute today
df['correct_signal'] = np.where(df['MA10'] > df['MA50'], 1, 0).shift(1)
correct_return = (df['correct_signal'] * df['daily_return']).sum()

print(f"❌ Biased return:  {biased_return:.2%}")   # Usually looks AMAZING
print(f"✅ Correct return: {correct_return:.2%}")  # Usually lower, but REAL
print(f"\nBias inflation: {biased_return - correct_return:.2%}")
# The difference is pure fantasy — it doesn't exist in reality!
💡 Survivorship Bias Fix in Practice

When backtesting on a group of stocks (like the S&P 500), always use historical index constituents, not today's current components. The Quandl/Sharadar dataset or CRSP in the US provides point-in-time membership data. For a beginner project, stick to a single well-known stock (AAPL, SPY) to avoid this issue entirely.

Reality Check

Transaction Cost Modeling: What Trades Actually Cost

Even a perfect, bias-free backtest can mislead you if it ignores the real costs of trading. Transaction costs are the various "frictions" that eat into your returns every time you buy or sell. They seem small — but for an active strategy, they can turn a winning system into a losing one.

🏠 Real Estate Analogy

Imagine buying and selling houses. Even if you perfectly time the market (buy low, sell high), you also pay: agent commission (2-3%), title fees, taxes, inspection costs, etc. After all these costs, many "profitable" house flips actually lose money. Trading has similar frictions. Ignoring them in your backtest is like doing your real estate math without any of those fees.

The Four Components of Transaction Costs

Transaction Cost Anatomy — What Happens When You Place an Order
💰
Commission
A fee charged by your broker per trade. Most retail brokers now charge $0 (Robinhood, IBKR Lite). Institutional brokers charge 0.01-0.05 cents per share. Always matters in high-frequency strategies.
↔️
Bid-Ask Spread
The difference between the price you can buy at (ask) and sell at (bid). For liquid stocks like AAPL it's ~$0.01. For small-cap stocks it can be 1-5%. You pay this every time you enter AND exit a trade.
🌊
Slippage
The difference between the price when you decide to trade and the price you actually get. By the time your order reaches the market, prices may have moved. Worse in volatile, illiquid markets.
🌍
Market Impact
Large orders move the market against you. Buying 1 million shares of a stock pushes the price up as you buy. You pay more than the initial price. Relevant for institutional (large) traders.

Transaction Cost Models: Simple to Complex

You can model costs at different levels of sophistication. Start simple, add complexity as needed:

Model 1: Fixed Cost Per Trade (Simplest)

Fixed Cost Model
Net Return per trade = Gross Return − Cost_per_trade / Position_Value
Best for: simple strategies with infrequent trades. Use when broker charges $0 commission and spread is negligible (large-cap stocks).

Model 2: Proportional Cost (Most Common)

Proportional Cost Model
Net Return = Gross Return − (spread/2 + commission) × 2 × turnover_rate
For each round trip (buy + sell), you pay spread/2 twice plus commission twice. If spread = 0.02% and commission = 0.01%, total round-trip cost = 0.06%.

Model 3: Square-Root Market Impact (Advanced)

Almgren-Chriss Market Impact
Impact = σ × √(Q / V_daily) × sign(Q)
σ = daily volatility, Q = order size, V_daily = average daily volume. This captures that large orders have non-linear cost — twice the size costs more than twice as much.
PYTHON — TRANSACTION COST MODELS 
import numpy as np
import pandas as pd

def apply_transaction_costs(returns, positions,
                             commission=0.001,   # 0.1% of trade value
                             spread=0.0002):    # 0.02% (2 bps) half-spread
    """
    Apply realistic transaction costs to backtest returns.
    
    Parameters:
    -----------
    returns   : pd.Series — daily gross returns
    positions : pd.Series — daily position (1=long, 0=flat, -1=short)
    commission: float     — commission as fraction of trade value
    spread    : float     — half bid-ask spread as fraction
    """
    # Detect trades: when position changes, a trade occurred
    trades = positions.diff().abs()  # Magnitude of position change
    
    # Total cost per trade = commission + spread (paid on entry AND exit)
    total_cost_per_unit = commission + spread
    
    # Cost = cost_per_unit × size of trade
    cost_series = trades * total_cost_per_unit
    
    # Net return = gross return − cost
    net_returns = returns - cost_series
    
    return net_returns, cost_series

# Example usage with our MA crossover strategy from Tab 2:
gross_returns = df['position'] * df['market_return']
net_returns, costs = apply_transaction_costs(
    gross_returns, df['position'],
    commission=0.0,    # Zero commission (retail broker)
    spread=0.00005    # 0.5 bps half-spread for liquid stock
)

print(f"Total trades: {(df['position'].diff().abs() > 0).sum()}")
print(f"Total cost drag: {costs.sum():.4%}")
print(f"Gross Sharpe: {gross_returns.mean()/gross_returns.std()*252**0.5:.2f}")
print(f"Net Sharpe:   {net_returns.mean()/net_returns.std()*252**0.5:.2f}")

⚠️ The Turnover Problem

Suppose your strategy makes 1,000 trades per year. Even at a tiny 0.05% total cost per round-trip, your annual cost drag = 1,000 × 0.05% = 50% per year. Your strategy needs to make >50% just to break even! This is why high-frequency strategies require either very low costs or very high edge per trade.

Gold Standard Validation

Walk-Forward Analysis: Truly Proving Your Strategy Works

Walk-Forward Analysis (WFA) is the most rigorous method to validate a trading strategy. It simulates how you would actually develop and deploy a strategy: train it on a window of past data, optimize parameters, then test on the immediately following out-of-sample period — then repeat, rolling forward through time.

🎓 Exam Analogy

Imagine a student who studies for 3 months, then takes an exam on material they've never seen — and repeats this process 10 times. If they consistently score well on each unseen exam, we can be confident they actually understand the material — not just memorized one past paper. That's walk-forward analysis: repeatedly testing on unseen data windows.

The Core Concept: Train-Test Windows Rolling Through Time

Full Walk-Forward Implementation

PYTHON — COMPLETE WFA IMPLEMENTATION 
import yfinance as yf
import pandas as pd
import numpy as np
import matplotlib.pyplot as plt
from itertools import product

# ── LOAD DATA ──────────────────────────────────────────────────
df = yf.download('SPY', start='2015-01-01', end='2023-12-31')
df['return'] = df['Close'].pct_change()
df = df.dropna()

def run_strategy(data, fast, slow):
    """Compute MA crossover strategy returns for given parameters."""
    ma_fast = data['Close'].rolling(fast).mean()
    ma_slow = data['Close'].rolling(slow).mean()
    signal  = (ma_fast > ma_slow).astype(float).shift(1)  # No look-ahead!
    ret = signal * data['return']
    return ret.mean() / ret.std() * 252**0.5  # Sharpe

def optimize_params(train_data):
    """Grid search for best MA parameters on training set."""
    fast_range = [5, 10, 15, 20]
    slow_range = [30, 40, 50, 60, 80]
    best_sharpe = -np.inf
    best_params = (10, 50)
    for fast, slow in product(fast_range, slow_range):
        sharpe = run_strategy(train_data, fast, slow)
        if sharpe > best_sharpe:
            best_sharpe, best_params = sharpe, (fast, slow)
    return best_params

# ── WALK-FORWARD ANALYSIS ──────────────────────────────────────
TRAIN_MONTHS = 18   # 18 months training
TEST_MONTHS  = 3    # 3 months testing (OOS)
STEP_MONTHS  = 3    # Step forward 3 months each iteration

oos_results = []   # Out-of-sample returns
windows     = []   # Track each window's metadata

# Convert months to trading days (approx 21 days/month)
TRAIN_DAYS = TRAIN_MONTHS * 21
TEST_DAYS  = TEST_MONTHS  * 21
STEP_DAYS  = STEP_MONTHS  * 21

start = 0
while start + TRAIN_DAYS + TEST_DAYS <= len(df):
    # Slice train and test windows
    train = df.iloc[start : start + TRAIN_DAYS]
    test  = df.iloc[start + TRAIN_DAYS : start + TRAIN_DAYS + TEST_DAYS]
    
    # Optimize on training data ONLY
    best_fast, best_slow = optimize_params(train)
    
    # Evaluate on test data using those optimized params
    ma_fast = test['Close'].rolling(best_fast).mean()
    ma_slow = test['Close'].rolling(best_slow).mean()
    signal  = (ma_fast > ma_slow).astype(float).shift(1)
    oos_ret = signal * test['return']
    
    oos_results.append(oos_ret)
    windows.append({
        'start': train.index[0], 'end': test.index[-1],
        'best_fast': best_fast, 'best_slow': best_slow
    })
    start += STEP_DAYS

# ── COMBINE ALL OOS RESULTS ────────────────────────────────────
all_oos = pd.concat(oos_results)
oos_sharpe = all_oos.mean() / all_oos.std() * 252**0.5
print(f"WFA Out-of-Sample Sharpe: {oos_sharpe:.2f}")
print(f"Number of windows tested: {len(oos_results)}")

# Plot OOS equity curve
(1 + all_oos).cumprod().plot(title='Walk-Forward Out-of-Sample Equity Curve')
plt.show()

Interpreting Walk-Forward Results

After running your WFA, you need to interpret what the results mean. Look for these signals:

📊
WFA Efficiency Ratio
OOS Sharpe / IS Sharpe. A ratio > 0.5 is generally good. Means your OOS performance is at least half of your in-sample performance. Values near 1.0 indicate minimal overfitting.
📉
Parameter Stability
Do the optimal parameters change wildly from window to window? If so, the strategy is too sensitive to the specific training period (overfitting). Stable parameters across windows = robust strategy.
🔄
Consistency
Is OOS performance positive across MOST windows, or just some? A strategy that wins 7 out of 10 windows is more trustworthy than one that wins 1 window by a huge amount.
🌡️
Regime Sensitivity
Does the strategy fail in specific market regimes (e.g., 2020 COVID crash, 2022 bear market)? This is important information — it tells you when to stop trading the strategy.
WFA Efficiency Ratio (Key Metric)
WFA Efficiency = OOS_Sharpe / IS_Sharpe × 100%
Example: IS Sharpe = 2.0, OOS Sharpe = 0.8 → Efficiency = 40%. This means 40% of in-sample performance persists out-of-sample. Values below 25% suggest severe overfitting. Values above 75% suggest a genuinely robust strategy.
✅ Walk-Forward vs Simple Train-Test Split

A simple train-test split (80% train, 20% test) only gives you ONE test period. That test period might happen to be unusual (e.g., just a bull market). Walk-forward analysis gives you MANY test periods across different market regimes — giving you much higher confidence in your results. This is why it's the industry gold standard.

📋 Quick Reference Cheat Sheet

📌 Backtesting Checklist — Use Before Every Backtest

□ Signal shifted by 1 period before computing returns (.shift(1))?
□ Using point-in-time data (no future data in the feature computation)?
□ Normalizer/scaler fit ONLY on training data, applied to test data?
□ Universe defined at each point in time (no survivorship bias)?
□ Transaction costs included (commission + spread + slippage)?
□ Out-of-sample period reserved and never touched during optimization?
□ Walk-forward validation run (not just a single train-test split)?
□ WFA Efficiency Ratio computed and > 40%?
□ Parameter stability checked across WFA windows?
□ Results interpreted relative to appropriate benchmark?
