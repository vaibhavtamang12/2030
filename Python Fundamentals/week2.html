Core Concepts You Must Know

📦 Variables & Data Types

Python variables are names that point to values. Types: int, float, str, bool, list, dict, tuple, set. Python is dynamically typed — the type is figured out at runtime, not declaration time.

⚙️ Functions

Functions are reusable blocks of code defined with def. They can take arguments (inputs) and return values. Functions are first-class objects in Python — you can pass them around like variables. This is critical for decorators.

🔁 Loops & Conditionals

for loops iterate over sequences. while loops run until a condition is False. if/elif/else handles branching. The in keyword checks membership. break exits a loop; continue skips to next iteration.

📚 Lists, Dicts, Sets, Tuples

The 4 main built-in containers. list: ordered, mutable. dict: key-value pairs. set: unordered unique values. tuple: ordered, immutable. List comprehensions: [x*2 for x in lst] — very common in Python.

📂 Modules & Imports

Code is organized in files called modules. Use import math, from math import sqrt, or import numpy as np. Python's standard library has hundreds of built-in modules. Third-party ones are installed via pip.

⚠️ Exception Handling

try/except/finally blocks let you handle errors gracefully. raise lets you throw your own errors. This is the foundation for context managers and professional code. Always handle exceptions in production quant code.

🔍 Scope (LEGB Rule)

Python looks up variables in this order: Local → Enclosing → Global → Built-in. This is crucial for understanding closures and decorators. global and nonlocal keywords modify scope rules.

🎯 First-Class Functions

In Python, functions are objects. You can store them in variables: f = print. Pass them as arguments: map(str, [1,2,3]). Return them from other functions. This is the foundation of decorators and higher-order functions.

// TOPIC 01 — OBJECT ORIENTED PROGRAMMING

OOP & Classes

OOP is a way to organize your code around things (objects) rather than just actions (functions). A class is a blueprint. An object is something built from that blueprint. Think: a Trade class is the blueprint; a specific Apple stock purchase is an object (instance) of that class.

💡 ANALOGY — FINANCIAL INSTRUMENT

Imagine you work at a trading desk. Every trade has the same structure: a symbol, price, quantity, direction. Instead of writing separate variables for each trade, you create a Trade class — a template. Every time you execute a trade, you create a new instance (object) of that template. The class is the blueprint; each trade execution is an object. Like a factory that stamps out Trade objects, each with their own data but following the same structure.

The 4 Pillars of OOP

📦 Encapsulation

Bundle data (attributes) and behavior (methods) together inside a class. Hide internal implementation details. Other code interacts through a clean interface. Like a black box: you use a Portfolio without knowing how it calculates NAV internally.

🧬 Inheritance

A child class can inherit all properties and methods of a parent class, then add or override behavior. EquityTrade inherits from Trade and adds equity-specific logic. Avoids repeating code.

🎭 Polymorphism

Different classes can respond to the same method call differently. trade.execute() works whether trade is an EquityTrade or OptionTrade — each implements it differently but the interface is the same.

🔒 Abstraction

Expose only what's necessary. Hide complexity. A RiskEngine class exposes .calculate_var(). Users don't need to know it runs 10,000 Monte Carlo simulations internally.

PYTHONstep3_methods.py — Adding behavior
class Trade:
    def __init__(self, symbol, price, qty, direction="BUY"):
        self.symbol    = symbol
        self.price     = price
        self.qty       = qty
        self.direction = direction
        self.is_open   = True

    # INSTANCE METHOD — operates on one specific trade object
    def calculate_pnl(self, current_price):
        """Calculate P&L of this trade."""
        raw_pnl = (current_price - self.price) * self.qty
        if self.direction == "SELL":
            raw_pnl = -raw_pnl   # reverse for short trades
        return raw_pnl

    def close_trade(self, close_price):
        """Close the position and report P&L."""
        pnl = self.calculate_pnl(close_price)
        self.is_open = False
        return {"symbol": self.symbol, "pnl": pnl}

    # CLASS METHOD — works on the class itself, not an instance
    @classmethod
    def from_order_dict(cls, order):
        """Alternative constructor — create Trade from a dict."""
        return cls(order["symbol"], order["price"], order["qty"])

    # STATIC METHOD — utility, no access to self or cls
    @staticmethod
    def validate_symbol(symbol):
        """Check if symbol is valid ticker format."""
        return symbol.isalpha() and symbol.isupper()


# ── Usage ─────────────────────────────────────────────────────
t = Trade("AAPL", 150.0, 100)
print(t.calculate_pnl(160.0))   # 1000.0
print(t.close_trade(155.0))     # {'symbol': 'AAPL', 'pnl': 500.0}

order = {"symbol": "GOOGL", "price": 2800, "qty": 10}
t2 = Trade.from_order_dict(order)   # classmethod usage
print(Trade.validate_symbol("AAPL"))  # True
Class vs Instance Attributes
ℹ️
Critical Distinction: Class attributes are shared by ALL instances. Instance attributes are unique to each object. Confusing these causes nasty bugs in production code.
PYTHONclass_vs_instance_attrs.py
class Portfolio:
    # CLASS ATTRIBUTE — shared by ALL Portfolio instances
    exchange = "NYSE"
    trade_count = 0       # danger: mutable class attribute!
    positions = []         # ⚠️ DANGER: shared list between instances!

    def __init__(self, name):
        # INSTANCE ATTRIBUTE — unique to each Portfolio
        self.name = name
        self.positions = []    # ✅ SAFE: each instance gets its own list
        Portfolio.trade_count += 1

p1 = Portfolio("Alpha Fund")
p2 = Portfolio("Beta Fund")

# Class attribute — same for all
print(p1.exchange)     # NYSE
print(p2.exchange)     # NYSE
print(Portfolio.trade_count)  # 2

# Instance attribute — different for each
print(p1.name)  # Alpha Fund
print(p2.name)  # Beta Fund

// TOPIC 02 — INHERITANCE & POLYMORPHISM

Inheritance

Inheritance lets you build new classes that extend existing ones. A child class gets everything the parent has, and can add new stuff or change existing behavior. This is how you build a hierarchy of financial instruments without repeating yourself.

💡 ANALOGY — INSTRUMENT HIERARCHY

In finance, all instruments share common properties: symbol, price, currency. But an Equity adds dividends and market cap. An Bond adds coupon rate and maturity. An Option adds strike, expiry, and Greeks. Inheritance lets you model this exactly: a base Instrument class, with specialized subclasses. You write common logic once. Subclasses inherit it for free, and only add what makes them special.

Basic Inheritance Syntax
PYTHONinheritance_basics.py
# ── PARENT CLASS ────────────────────────────────────────────────
class Instrument:
    """Base class for all financial instruments."""

    def __init__(self, symbol, price, currency="USD"):
        self.symbol   = symbol
        self.price    = price
        self.currency = currency

    def get_price(self):
        return f"{self.symbol}: {self.price} {self.currency}"

    def to_dict(self):
        return {"symbol": self.symbol, "price": self.price}


# ── CHILD CLASS ──────────────────────────────────────────────────
# Equity IS-A Instrument — it gets everything Instrument has
class Equity(Instrument):    # ← parentheses declare inheritance
    """Stock/equity instrument."""

    def __init__(self, symbol, price, dividend_yield=0.0):
        # super() calls parent's __init__ — DON'T repeat parent's init code!
        super().__init__(symbol, price)
        self.dividend_yield = dividend_yield   # equity-specific attribute

    # NEW method — only on Equity, not on Instrument
    def annual_dividend(self):
        return self.price * self.dividend_yield

    # OVERRIDE parent method — polymorphism in action
    def to_dict(self):
        d = super().to_dict()          # get parent's dict
        d["dividend_yield"] = self.dividend_yield  # extend it
        return d


class Bond(Instrument):
    def __init__(self, symbol, price, coupon_rate, maturity_years):
        super().__init__(symbol, price)
        self.coupon_rate    = coupon_rate
        self.maturity_years = maturity_years

    def annual_coupon(self):
        return self.price * self.coupon_rate

    def to_dict(self):
        d = super().to_dict()
        d["coupon_rate"] = self.coupon_rate
        return d


# ── USAGE ────────────────────────────────────────────────────────
aapl  = Equity("AAPL", 175.0, 0.005)
tbill = Bond("US10Y", 98.5, 0.045, 10)

print(aapl.get_price())     # AAPL: 175.0 USD  ← inherited method
print(aapl.annual_dividend())  # 0.875  ← child method

print(tbill.get_price())   # US10Y: 98.5 USD  ← inherited
print(tbill.annual_coupon()) # 4.4325  ← child method

# isinstance checks — true up the hierarchy
print(isinstance(aapl, Equity))      # True
print(isinstance(aapl, Instrument))   # True  ← inheritance!
print(isinstance(aapl, Bond))         # False

# Polymorphism: treat different objects through the same interface
# The 'print_report' function doesn't care WHAT type of instrument it gets
# It just calls .to_dict() and .get_price() — works on ALL of them!

def print_report(instruments):
    """Works with ANY Instrument subclass — polymorphism!"""
    print("=" * 40)
    for instr in instruments:
        print(instr.get_price())        # calls each class's version
        print(instr.to_dict(), "\n")   # each class overrides to_dict()

Abstract Base Classes (ABCs)
⚠️
Abstract Classes: Sometimes you want to force every subclass to implement certain methods. Use abc.ABC and @abstractmethod. Any class that inherits but doesn't implement the abstract methods will raise a TypeError when instantiated.
PYTHONabstract_classes.py — Enforcing a contract
from abc import ABC, abstractmethod

class TradingStrategy(ABC):
    """Abstract base: every strategy MUST implement these methods."""

    @abstractmethod
    def generate_signal(self, market_data) -> str:
        """Must return 'BUY', 'SELL', or 'HOLD'."""
        ...   # body doesn't matter — it's abstract

    @abstractmethod
    def calculate_position_size(self, signal, portfolio_value) -> float:
        ...

    # Concrete method — inherited as-is by all subclasses
    def describe(self):
        return f"Strategy: {type(self).__name__}"


class MomentumStrategy(TradingStrategy):
    def __init__(self, lookback=20):
        self.lookback = lookback

    def generate_signal(self, market_data):
        prices = market_data["close"][-self.lookback:]
        if prices[-1] > prices[0]:
            return "BUY"
        elif prices[-1] < prices[0]:
            return "SELL"
        return "HOLD"

    def calculate_position_size(self, signal, portfolio_value):
        return portfolio_value * 0.1 if signal != "HOLD" else 0


# TradingStrategy()        ← TypeError: Can't instantiate abstract class
strat = MomentumStrategy() # ✅ OK — all abstract methods are implemented
print(strat.describe())    # Strategy: MomentumStrategy
Method Resolution Order (MRO)

When Python looks up a method, it follows a specific order through the inheritance tree. You can inspect it:

PYTHONmro.py
class A:
    def hello(self): print("A")

class B(A):
    def hello(self): print("B")

class C(A):
    def hello(self): print("C")

class D(B, C):  # Multiple inheritance
    pass

d = D()
d.hello()          # B — Python uses C3 linearization
print(D.__mro__)    # (D, B, C, A, object)
# Left to right, depth first, but each class appears only once

// TOPIC 03 — DUNDER / MAGIC METHODS

__dunder__ Methods

Dunder methods (double underscore methods, aka "magic methods") are special methods Python calls automatically in certain situations. When you write a + b, Python calls a.__add__(b). When you do len(x), Python calls x.__len__(). They let your custom classes integrate seamlessly with Python's syntax.

💡 ANALOGY — THE PROTOCOL SYSTEM

Think of dunder methods as contracts with Python. Python says: "If you implement __len__, I'll let you use len() on your object. If you implement __add__, I'll let people write obj1 + obj2." By implementing these contracts, your Portfolio class can behave just like a built-in Python type — you can iterate it, add them together, compare them, print them nicely. It's the difference between a class that works and one that feels native.

Building a Full-Featured Portfolio Class
PYTHONportfolio_dunders.py — All major dunders demonstrated
from functools import total_ordering

@total_ordering   # fills in all comparison methods from __eq__ + __lt__
class Portfolio:
    """A portfolio that behaves like a native Python container."""

    def __init__(self, name, positions=None):
        self.name      = name
        self._positions = positions or {}   # {symbol: (qty, avg_price)}

    # ── REPRESENTATION ──────────────────────────────────────────────
    def __repr__(self):
        """Unambiguous — useful in debugger/REPL."""
        return f"Portfolio(name={self.name!r}, positions={len(self._positions)})"

    def __str__(self):
        """Human-readable — used by print()."""
        lines = [f"Portfolio: {self.name}"]
        for sym, (qty, price) in self._positions.items():
            lines.append(f"  {sym}: {qty} @ {price:.2f}")
        return "\n".join(lines)

    # ── CONTAINER PROTOCOL ──────────────────────────────────────────
    def __len__(self):
        """len(portfolio) → number of positions."""
        return len(self._positions)

    def __getitem__(self, symbol):
        """portfolio['AAPL'] → returns position data."""
        if symbol not in self._positions:
            raise KeyError(f"No position for {symbol}")
        return self._positions[symbol]

    def __setitem__(self, symbol, value):
        """portfolio['AAPL'] = (100, 150.0)"""
        self._positions[symbol] = value

    def __contains__(self, symbol):
        """'AAPL' in portfolio → True/False."""
        return symbol in self._positions

    def __iter__(self):
        """for symbol in portfolio → iterate symbols."""
        return iter(self._positions)

    # ── ARITHMETIC ──────────────────────────────────────────────────
    def __add__(self, other):
        """portfolio1 + portfolio2 → combined portfolio."""
        combined = dict(self._positions)
        for sym, (qty, price) in other._positions.items():
            if sym in combined:
                # Merge positions with weighted average price
                old_qty, old_price = combined[sym]
                new_qty = old_qty + qty
                new_price = (old_qty * old_price + qty * price) / new_qty
                combined[sym] = (new_qty, new_price)
            else:
                combined[sym] = (qty, price)
        return Portfolio(f"{self.name}+{other.name}", combined)

    def __mul__(self, scalar):
        """portfolio * 2 → scale all quantities."""
        scaled = {sym: (int(qty * scalar), price)
                  for sym, (qty, price) in self._positions.items()}
        return Portfolio(f"{self.name}×{scalar}", scaled)

    # ── COMPARISON ──────────────────────────────────────────────────
    def total_value(self):
        return sum(qty * price for qty, price in self._positions.values())

    def __eq__(self, other):
        """Two portfolios are equal if they have same value."""
        return self.total_value() == other.total_value()

    def __lt__(self, other):
        return self.total_value() < other.total_value()

    # ── CALLABLE ────────────────────────────────────────────────────
    def __call__(self, symbol):
        """portfolio('AAPL') — same as portfolio['AAPL']."""
        return self[symbol]


# ── USAGE ────────────────────────────────────────────────────────
p1 = Portfolio("Alpha", {"AAPL": (100, 175.0), "MSFT": (50, 380.0)})
p2 = Portfolio("Beta",  {"GOOGL": (20, 2800.0)})

print(p1("AAPL"))     # (100, 175.0) — calls __call__

    print(sym)

__repr__ vs __str__ — Know The Difference
✅
Rule of thumb: __repr__ should be unambiguous and ideally let you recreate the object: Trade('AAPL', 150.0, 100). __str__ should be readable by a human: "AAPL: 100 shares at $150.00". If only __repr__ is defined, Python uses it for both. In the debugger and REPL, __repr__ is what you see.

// TOPIC 04 — DECORATORS

Decorators

A decorator is a function that wraps another function to add behavior before, after, or around it — without modifying the original function's code. They are built on the principle that functions are first-class objects (from our prerequisites). The @decorator syntax is just beautiful shorthand for func = decorator(func).

💡 ANALOGY — THE TRADING DESK STAMP

Imagine every trade that goes through your desk needs to be: logged, risk-checked, and timed. Instead of adding this boilerplate to every single trade function, you write it once as a decorator and stamp it on any function you want. The function itself stays clean and focused. The decorator handles the cross-cutting concerns. This is exactly what production quant systems do — @log_trade, @risk_check, @timer decorators on every execution function.

        return x * factor    # 'factor' is remembered from outer scope

PYTHONstep2_basic_decorator.py — First decorator
from functools import wraps
import time

# A decorator is a function that takes a function and returns a function
def timer(func):
    """Times how long a function takes to execute."""
    @wraps(func)   # preserves original function's name & docstring
    def wrapper(*args, **kwargs):   # *args, **kwargs: accept any arguments
        start  = time.perf_counter()
        result = func(*args, **kwargs)   # call the ORIGINAL function
        end    = time.perf_counter()
        print(f"{func.__name__} took {end-start:.4f}s")
        return result   # return original result unchanged
    return wrapper


# METHOD 1: Explicit decoration
def run_backtest():
    time.sleep(0.1)
    return 0.15

run_backtest = timer(run_backtest)   # wrap it manually


# METHOD 2: @ syntax — EXACTLY the same as above, just prettier
@timer
def run_backtest_v2():
    time.sleep(0.1)
    return 0.15

result = run_backtest_v2()   # run_backtest_v2 took 0.1002s
print(result)               # 0.15  ← original return value preserved

from functools import wraps

# Decorator WITH arguments needs an extra layer of nesting
def retry(max_attempts=3, delay=1.0, exceptions=(Exception,)):
    """Retry a function up to max_attempts times on failure."""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(max_attempts):
                try:
                    return func(*args, **kwargs)
                except exceptions as e:
                    if attempt == max_attempts - 1:
                        raise   # re-raise on final attempt
                    print(f"Attempt {attempt+1} failed: {e}. Retrying...")
                    time.sleep(delay)
        return wrapper
    return decorator

import logging, time, functools
from functools import wraps

# 1. LOGGING DECORATOR ───────────────────────────────────────────
def log_trade(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        logging.info(f"TRADE: {func.__name__} called with args={args} kwargs={kwargs}")
        result = func(*args, **kwargs)
        logging.info(f"TRADE: {func.__name__} returned {result}")
        return result
    return wrapper

# 2. RISK CHECK DECORATOR ────────────────────────────────────────
def risk_check(max_position_size=1_000_000):
    def decorator(func):
        @wraps(func)
        def wrapper(symbol, qty, price, *args, **kwargs):
            notional = qty * price
            if notional > max_position_size:
                raise ValueError(
                    f"Position {symbol}: notional {notional:,.0f} exceeds limit {max_position_size:,.0f}"
                )
            return func(symbol, qty, price, *args, **kwargs)
        return wrapper
    return decorator

# 3. MEMOIZE (manual cache) ──────────────────────────────────────
def memoize(func):
    cache = {}
    @wraps(func)
    def wrapper(*args):
        if args not in cache:
            cache[args] = func(*args)
        return cache[args]
    wrapper.cache = cache    # expose cache for inspection
    return wrapper

# 4. STACKING DECORATORS ─────────────────────────────────────────
# Applied bottom-up: risk_check runs first, then log_trade, then the function

class Position:
    def __init__(self, symbol, qty, price):
        self.symbol = symbol
        self.qty    = qty
        self._price = price   # underscore = "private by convention"

    @property
    def price(self):
        """Getter — called when you access position.price"""
        return self._price

    @price.setter
    def price(self, value):
        """Setter — called when you do position.price = x"""
        if value <= 0:
            raise ValueError("Price must be positive")
        self._price = value

    @property
    def market_value(self):
        """Computed property — calculated on access, no setter."""
        return self.qty * self._price

// TOPIC 05 — GENERATORS

Generators

A generator is a special function that lazily produces values one at a time, instead of building an entire list in memory. It uses yield instead of return. The magic: execution pauses at each yield and resumes where it left off when you ask for the next value. Perfect for streaming huge amounts of market data.

💡 ANALOGY — TICK DATA STREAM

Imagine you have 10 years of tick-by-tick market data — billions of rows. If you load it all into a list, you'll crash your computer (it won't fit in RAM). A generator is like a fire hose that you control: it produces one tick at a time, you process it, then ask for the next one. The data is never all in memory at once. In live trading, your price feed IS a generator — it produces the next price only when available. Generators are also the foundation of Python's async/await concurrency model.

yield vs return — The Core Difference
PYTHONgenerators_basics.py
# STEP 1: Your first generator
def countdown(n):
    print("Starting countdown...")
    while n > 0:
        yield n       # pause here, give n to caller
        n -= 1        # resumes here when next() is called
    print("Done!")

gen = countdown(3)    # NOTHING RUNS YET — just creates generator object
print(gen)            # <generator object countdown at 0x...>

print(next(gen))   # Starting countdown...
                   # 3
print(next(gen))   # 2    ← resumes from where it left off
print(next(gen))   # 1
# print(next(gen))   ← raises StopIteration: generator exhausted

# Generators work in for loops naturally (for calls next() automatically)
for n in countdown(5):
    print(n, end=" ")   # 5 4 3 2 1

def price_feed(symbol, start_price, volatility=0.01):
    """Infinite generator simulating a live price stream."""
    price = start_price
    while True:
        change = random.gauss(0, volatility)
        price *= (1 + change)
        yield {"symbol": symbol, "price": round(price, 4),
               "timestamp": datetime.now().isoformat()}

def candlestick_generator(csv_path, chunk_size=1000):
    """Read massive OHLCV file in chunks — never loads all at once."""
    with open(csv_path) as f:
        headers = f.readline().strip().split(",")
        chunk = []
        for line in f:
            row = dict(zip(headers, line.strip().split(",")))
            chunk.append(row)
            if len(chunk) == chunk_size:
                yield chunk
                chunk = []
        if chunk:   # yield remaining
            yield chunk

def rolling_window(data, window):
    """Generate sliding windows of 'window' size."""
    for i in range(len(data) - window + 1):
        yield data[i:i + window]

prices = [100, 102, 101, 103, 105, 104]

def generate_prices(n): yield from (100 + random.gauss(0, 2) for _ in range(n))

        if abs(p - 100) < threshold: yield p
def scale_prices(prices, factor):
    yield from (p * factor for p in prices)

def flatten(nested):
    for item in nested:
        if isinstance(item, list):
            yield from flatten(item)   # recursive!
        else:
            yield item

print(list(flatten(sectors)))  # ['AAPL', 'MSFT', 'JPM', 'GS', 'MS']

// TOPIC 06 — CONTEXT MANAGERS

Context Managers

A context manager is an object that defines what happens when you enter and exit a with block. They guarantee cleanup code always runs — even if an exception occurs. Perfect for managing resources: database connections, file handles, locks, and trading sessions that must be properly opened and closed.

💡 ANALOGY — TRADING SESSION

Imagine opening a trading session at market open: you authenticate, set up connections, initialize risk parameters. At market close, you must flatten all positions, disconnect, save state — no matter what happened during the day (errors, crashes, exceptions). A context manager is like a guaranteed cleanup contract. with trading_session() as session: — the session always opens and always closes properly, even if your strategy crashes halfway through.

The Problem Context Managers Solve
PYTHONthe_problem.py — Without context managers
# ❌ WITHOUT context manager — resource leak if exception occurs
def bad_approach():
    conn = database.connect()
    data = conn.query("SELECT * FROM trades")  # what if THIS throws?
    conn.close()   # NEVER REACHED if query fails!
    return data   # leaked connection!

# ❌ Manual try/finally — correct but ugly and repetitive
def manual_approach():
    conn = None
    try:
        conn = database.connect()
        data = conn.query("SELECT * FROM trades")
        return data
    except Exception as e:
        print(f"Error: {e}")
    finally:
        if conn:
            conn.close()  # always runs — correct but boilerplate

# ✅ WITH context manager — clean, guaranteed, Pythonic
def good_approach():
    with database.connect() as conn:  # opens on entry, closes on exit
        data = conn.query("SELECT * FROM trades")
        return data
    # conn.close() called automatically — ALWAYS — even if exception thrown

PYTHONmethod1_class.py — Class-based context manager
import time

class TradingSession:
    """Context manager for a trading session."""

    def __init__(self, session_name, risk_limit=1_000_000):
        self.session_name = session_name
        self.risk_limit   = risk_limit
        self.start_time   = None
        self.trades       = []

    def __enter__(self):
        """Called at start of 'with' block. Return value is bound to 'as' variable."""
        self.start_time = time.time()
        print(f"[SESSION] Opening: {self.session_name}")
        print(f"[SESSION] Risk limit: ${self.risk_limit:,}")
        return self    # this is what 'as session' refers to

    def __exit__(self, exc_type, exc_val, exc_tb):
        """Called at end of 'with' block. 3 args: exception info (None if no error)."""
        elapsed = time.time() - self.start_time
        print(f"[SESSION] Closing: {self.session_name}")
        print(f"[SESSION] Duration: {elapsed:.2f}s, Trades: {len(self.trades)}")

        if exc_type is not None:
            # An exception occurred in the 'with' block
            print(f"[SESSION] ⚠️ Error: {exc_val} — rolling back trades!")
            self.trades.clear()

        # return True to suppress exception, False/None to re-raise it
        return False   # don't suppress — let error propagate

    def execute_trade(self, symbol, qty, price):
        self.trades.append({"symbol": symbol, "qty": qty, "price": price})
        print(f"  Trade: {qty} {symbol} @ {price}")


# ── USAGE ──────────────────────────────────────────────────────

    session.execute_trade("AAPL", 100, 175.0)
    session.execute_trade("MSFT", 50, 380.0)
    # session closes automatically here, prints summary

from contextlib import contextmanager
import time

# The @contextmanager decorator turns a generator function into a context manager
# Code BEFORE yield = __enter__
# The yield value = what 'as' binds to
# Code AFTER yield = __exit__

@contextmanager
def timer_context(label):
    """Context manager that times a block of code."""
    start = time.perf_counter()
    print(f"⏱  Starting: {label}")
    try:
        yield           # body of 'with' block runs here
    finally:            # always runs, even on exception
        elapsed = time.perf_counter() - start
        print(f"⏱  {label} done in {elapsed:.4f}s")


@contextmanager
def managed_position(symbol, risk_system):
    """Open a position, ensure it's always tracked/closed."""
    risk_system.register(symbol)
    try:
        position = {"symbol": symbol, "qty": 0, "open": True}
        yield position
    except Exception as e:
        print(f"Error in position {symbol}: {e}")
        raise
    finally:
        position["open"] = False
        risk_system.unregister(symbol)
        print(f"Position {symbol} closed and unregistered from risk")

    console = logging.StreamHandler(sys.stdout)
    console.setLevel(logging.WARNING)
    console.setFormatter(fmt)
    logger.addHandler(console)

    # ── HANDLER 2: Rotating file (all levels) ──────────────────────
    if log_file:
        Path(log_file).parent.mkdir(parents=True, exist_ok=True)
        rotating = logging.handlers.RotatingFileHandler(
            log_file,
            maxBytes=10 * 1024 * 1024,   # 10 MB per file
            backupCount=5               # keep 5 old files
        )
        rotating.setLevel(logging.DEBUG)
        rotating.setFormatter(fmt)
        logger.addHandler(rotating)

    return logger


# ── PER-MODULE LOGGER PATTERN ───────────────────────────────────
# Each module gets its own logger — standard Python practice
class TradingEngine:
    def __init__(self):
        # Logger named after class — creates hierarchy: root → TradingEngine
        self.logger = logging.getLogger(f"{__name__}.TradingEngine")

    def execute_trade(self, symbol, qty, price):
        self.logger.info(f"Executing trade: {symbol} qty={qty} price={price}")
        try:
            # ... trade logic ...
            self.logger.debug(f"Trade order sent to broker for {symbol}")
            return True
        except Exception as e:

            self.logger.critical("Initiating emergency position liquidation!")

// TOPIC 08 — PDB DEBUGGING

Python Debugger (pdb)

PDB is Python's built-in interactive debugger. Instead of adding 50 print() statements and re-running, you drop a breakpoint in your code and pause execution at that exact line. You then inspect all variables, step through code one line at a time, and understand exactly what's happening. An essential skill for finding bugs in complex trading algorithms.

💡 ANALOGY — SLOW MOTION REPLAY

Imagine your trading algorithm produced a wrong PnL calculation. With print() debugging, you're guessing and re-running. With PDB, it's like having a slow-motion replay of your code's execution. You pause at any line, look at every variable's exact value at that exact moment, step forward one instruction at a time, and run Python expressions to test hypotheses — all while the program is paused mid-execution. It's the difference between watching a blurry video and pausing on every frame.

3 Ways to Enter PDB
PYTHONentering_pdb.py — All 3 methods
# ── METHOD 1: breakpoint() — Python 3.7+ preferred way ─────────
def calculate_sharpe(returns, risk_free_rate=0.02):
    excess = [r - risk_free_rate/252 for r in returns]
    breakpoint()   # ← execution pauses HERE, drops into PDB
    mean  = sum(excess) / len(excess)
    std   = ((sum((x - mean)**2 for x in excess) / len(excess))**0.5)
    return (mean / std) * (252**0.5)

# ── METHOD 2: import pdb; pdb.set_trace() — older style ────────
import pdb

def risky_function(data):
    pdb.set_trace()    # same as breakpoint() but explicit
    return process(data)

# ── METHOD 3: Post-mortem on crash ──────────────────────────────
import pdb

try:
    result = calculate_sharpe([])   # ZeroDivisionError!
except:
    pdb.post_mortem()  # drops into PDB at the point of failure

# ── METHOD 4: Command line — debug entire script ────────────────
# Run from terminal: python -m pdb my_strategy.py
# Pauses at first line. Use 'c' to continue to first breakpoint

# BUGGY CODE — Sharpe ratio is coming out wrong. Let's debug it.

def compute_returns(prices):
    """Compute daily returns from price series."""
    return [(prices[i] - prices[i-1]) / prices[i-1]
            for i in range(1, len(prices))]

def sharpe_ratio(prices, risk_free=0.02):
    """Annualized Sharpe ratio — BUGGY!"""
    returns = compute_returns(prices)
    breakpoint()    # ← Let's investigate here
    daily_rf = risk_free / 252
    excess_returns = [r - daily_rf for r in returns]
    mean  = sum(excess_returns) / len(excess_returns)
    # BUG: forgot to divide by N before taking sqrt → wrong std!
    variance = sum((x - mean)**2 for x in excess_returns)
    std  = variance ** 0.5   # ← BUG IS HERE (should be / N)
    return (mean / std) * (252 ** 0.5)

prices = [100, 101, 100.5, 102, 101.5, 103]
print(sharpe_ratio(prices))

import pdb

# ── CONDITIONAL BREAKPOINT ──────────────────────────────────────
# Only pause when a specific condition is true
def process_trades(trades):
    for i, trade in enumerate(trades):
        if trade["qty"] > 1000:
            breakpoint()   # only pauses for large trades
        process(trade)

# ── PDB COMMAND LINE BREAKPOINTS ────────────────────────────────
# Inside pdb:
# b 25                  ← break at line 25
# b process_trades      ← break at function start
# b 25, qty > 1000      ← CONDITIONAL break at line 25
# commands 1            ← run commands when breakpoint 1 hits
# p trade               ← print trade each time
# end

# ── WATCHDOG DEBUGGING ──────────────────────────────────────────
# Watch an expression change
class Watchdog:
    """Drop into debugger when a value changes unexpectedly."""
    def __setattr__(self, name, value):
        if name == "pnl" and hasattr(self, "pnl"):
            if value < self.pnl - 10000:  # sudden large loss!
                breakpoint()
        super().__setattr__(name, value)
